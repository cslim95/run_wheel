# Run Reach Map — Build Brief

## What we're building

A web visualization that shows the furthest distance run from home in each 20° compass segment around the user's home location. The output is a map with:

- A pin at home
- 18 shaded segments showing the furthest-reach point in each 20° segment - darker for further
- All qualifying run tracks rendered faintly underneath (should show through underneath the shaded segments still)

The intuition: like fog-of-war in a game, this shows your personal "running territory" and reveals directional asymmetry (e.g. "I run south along the coast 5x further than I run north into the city").

## Architecture

Two parts:

1. **Backend (Python)** — extends an existing script called `run_explorer` that already fetches and caches Strava activities with resume-on-rate-limit behaviour. We extend it to fetch GPS streams for each activity, cache them, and export a single `runs.json` file.
2. **Frontend (static SPA)** — Vite + React + TypeScript + MapLibre. Reads `runs.json`, lets the user drop a pin to set their home location (saved to localStorage), then computes and renders the reach map client-side.

The frontend does **all** reach computation. This is intentional: moving the home pin re-computes everything instantly without re-running Python.

Deployment: frontend is a static SPA. For now it runs locally (`npm run dev`). Python runs locally; its output is written into the frontend's `public/` directory.

## Backend changes (`run_explorer`)

The existing Python script already:

- Authenticates with Strava
- Fetches the user's activities list via `/athlete/activities`
- Caches results to disk
- On hitting a 429 rate limit, saves state and exits cleanly so re-running resumes from where it stopped

### What to add

1. **Fetch GPS streams.** For each `Run` activity that doesn't already have a cached stream, call:
   ```
   GET https://www.strava.com/api/v3/activities/{id}/streams?keys=latlng&key_by_type=true
   ```
   Response shape:
   ```json
   { "latlng": { "data": [[-33.89, 151.27], [-33.891, 151.271], ...] } }
   ```
   Cache each stream to disk (one file per activity, or extend existing cache structure — match whatever pattern the existing code uses). Apply the same resume-on-429 behaviour: catch the rate-limit error, save state, exit. Re-running picks up at the next un-fetched activity.

2. **Add an export step.** A new command/function that walks the cache and writes `runs.json` to a configurable output path. Default the path to `../run-reach-map/public/runs.json`.

### `runs.json` schema

```json
{
  "generated_at": "2026-04-27T10:00:00Z",
  "athlete_id": 12345,
  "runs": [
    {
      "id": 8675309,
      "name": "Morning Run",
      "start_date": "2026-04-15T06:23:00Z",
      "distance_m": 10523,
      "moving_time_s": 2841,
      "start_latlng": [-33.89, 151.27],
      "coords": [[-33.89, 151.27], [-33.891, 151.271], ...]
    }
  ]
}
```

Filter rules for inclusion:
- `type == "Run", "Walk", "Trail Run", "Hike"`only (not Ride, Ski etc.)
- `coords.length >= 2` (skip degenerate activities)
- Activity must have a `start_latlng` (skip indoor treadmill activities)

### Constants

```ts
export const HOME_RADIUS_M = 500;     // qualifying origin radius
export const MIN_RUNS = 5;            // min qualifying runs for valid home
export const SEGMENT_COUNT = 18;      // 360 / 20
export const SEGMENT_DEGREES = 20;
```

### Core algorithm (`src/lib/reach.ts`)

```ts
import * as turf from '@turf/turf';

export interface Run {
  id: number;
  name: string;
  coords: [number, number][];  // [lat, lng]
  // ...
}

export interface ReachPoint {
  bearing_centre: number;      // degrees, e.g. 10, 30, 50, ... (segment midpoint)
  distance_m: number;
  coord: [number, number];     // [lat, lng] of the furthest point
  run_id: number;              // which run achieved this reach
}

export interface ReachResult {
  qualifyingRuns: Run[];       // runs that started within HOME_RADIUS_M of home
  reachPoints: ReachPoint[];   // up to SEGMENT_COUNT entries (some segments may be empty)
  totalRuns: number;
}

export function computeReach(runs: Run[], home: [number, number]): ReachResult {
  const homePt = turf.point([home[1], home[0]]); // turf uses [lng, lat]

  // 1. Filter to runs starting within HOME_RADIUS_M of home
  const qualifyingRuns = runs.filter(r => {
    if (!r.coords.length) return false;
    const [lat, lng] = r.coords[0];
    const start = turf.point([lng, lat]);
    return turf.distance(homePt, start, { units: 'meters' }) <= HOME_RADIUS_M;
  });

  if (qualifyingRuns.length < MIN_RUNS) {
    return { qualifyingRuns, reachPoints: [], totalRuns: runs.length };
  }

  // 2. For each segment, track the max-distance point
  const buckets: (ReachPoint | null)[] = Array(SEGMENT_COUNT).fill(null);

  for (const run of qualifyingRuns) {
    for (const [lat, lng] of run.coords) {
      const pt = turf.point([lng, lat]);
      const dist = turf.distance(homePt, pt, { units: 'meters' });
      let bearing = turf.bearing(homePt, pt); // -180 to 180
      if (bearing < 0) bearing += 360;        // normalise to 0–360
      const idx = Math.floor(bearing / SEGMENT_DEGREES) % SEGMENT_COUNT;

      if (!buckets[idx] || dist > buckets[idx]!.distance_m) {
        buckets[idx] = {
          bearing_centre: idx * SEGMENT_DEGREES + SEGMENT_DEGREES / 2,
          distance_m: dist,
          coord: [lat, lng],
          run_id: run.id,
        };
      }
    }
  }

  return {
    qualifyingRuns,
    reachPoints: buckets.filter((b): b is ReachPoint => b !== null),
    totalRuns: runs.length,
  };
}
```

### UI flow

**On first load:**
1. Fetch `/runs.json`.
2. Check localStorage for `home_location`.
3. If absent: enter "setup mode" — show the map zoomed to roughly fit all run start points (or default to Sydney CBD), with a prompt "Click the map to set your home location."
4. On map click in setup mode: save coordinates to localStorage, exit setup mode, render the reach map.

**Reach map mode — map layers (bottom to top):**
1. Base tiles
2. **Faded run tracks** — all qualifying runs as a line layer, opacity ~0.15, thin stroke (~1px). Use a single MapLibre line source with all run coords as a MultiLineString for performance.
3. **Reach polygon** — GeoJSON polygon connecting the reach points in order of bearing. Semi-transparent fill (~0.2 opacity), distinct stroke.
4. **Reach point markers** — small circles at each of the (up to) 18 max-distance points.
5. **Home pin** — distinctive marker at home location.

**UI controls (sidebar or top bar):**
- A "Reset home location" button that clears localStorage and re-enters setup mode.
- A stats panel showing:
  - Qualifying runs (e.g. "47 of 312 runs start within 500m of home")
  - Max reach + the name/date of the run that achieved it
  - Average reach across segments
  - Number of segments with data (e.g. "16 of 18 segments")

### Edge cases

- **Fewer than `MIN_RUNS` qualifying runs from this home:** show a message like "Only X of 5 runs found starting within 500m of this location. Try a different pin location, or upload more runs."
- **Empty segments:** some segments may have no qualifying GPS points (e.g. home on coast, segment points out to sea). The reach polygon should *skip* those segments rather than draw an awkward straight line across them. Easiest approach: split the polygon into multiple polygons wherever there's a gap, or render a polyline that breaks at gaps and only fills closed sections. If skipping turns out to be fiddly, drawing only the markers + connecting only adjacent non-empty segments with straight lines is an acceptable v1.
- **Coordinate order:** `runs.json` stores `[lat, lng]`. MapLibre and Turf both use `[lng, lat]`. Be consistent — convert at the boundary, ideally once when loading the JSON.

## Out of scope for v1 (but worth keeping the architecture friendly to)

- **Multi-user / sharing.** Currently just-for-me. The frontend already reads a single JSON file, so a future sharing version can either (a) add a serverless OAuth backend that produces the same JSON shape on demand, or (b) accept user-uploaded Strava bulk-export zips and parse GPX in-browser. Either way the reach computation, map rendering, and home-pin UX don't need to change.
- **Auto home detection.** v1 is drop-a-pin only. Later: geometric median of run starts, or DBSCAN for multi-home users.
- **Per-segment minimum-runs filter.** Currently only the origin filter (5+ runs from home) is applied. A future version might also require N runs contributing to a segment before showing it, to avoid one outlier race dominating.
- **Activity types beyond Run.** Walks, hikes, rides could be toggleable layers later.
